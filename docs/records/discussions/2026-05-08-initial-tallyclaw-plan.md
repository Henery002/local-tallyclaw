# TallyClaw 初始方案讨论

日期：2026-05-08  
状态：已记录

## 背景

项目所有者希望创建一个本地桌面像素宠物，形态上可参考 Codex app 内的宠物，但核心用途是观测本机工具和网关产生的 API token 调度与用量。

暂定项目名为 TallyClaw，本地项目目录为 `/Users/henery/code/local-tallyclaw`。

讨论中的主要源系统：

- cockpit tools API 服务统计。
- `local-ai-gateway` 用量或统计数据。
- 其他本地工具作为低优先级后续来源。

## 已同意方向

TallyClaw 应是一个本地、只读的 token 可观测宠物。

它应当：

- 读取已批准本地工具和网关中的既有 token 用量记录。
- 绝不写入上游工具或网关。
- 持久化自己的长期用量账本。
- 提供日、周、月和永久 token 统计。
- 第一版 UI 保持紧凑，并以宠物体验为中心。
- 如后续确有必要，再补充详细 dashboard。

## 重要安全边界

TallyClaw 不能修改任何源数据。

讨论中明确包含的例子：

- 不写入 cockpit tools 数据文件。
- 不刷新 cockpit tools 中 Codex 账号的 `refresh_token` 配置。
- 不写入 `local-ai-gateway` 数据文件。
- 不删除、重置、迁移、压缩或修复源 token 统计。

## 数据需求

目标颗粒度参考 cockpit tools API 服务观测面板：

- 总请求数。
- 成功和失败请求数。
- 总 token 数。
- 输入和输出 token 数。
- 可用时记录缓存 token 与思考 token。
- 平均延迟。
- 成功率。
- 日、周、月和永久总量。

永久总量应在 TallyClaw 完成记录后继续保留，即使上游 token 数据文件后续被删除也不回退。

## 推荐架构

第一版采用原生 macOS 应用：

- SwiftUI 负责应用 UI。
- AppKit 负责浮动、可拖动、常驻顶部窗口行为。
- SpriteKit 或轻量帧动画负责流畅像素宠物动效。
- SQLite 负责项目自有持久化。
- 数据源适配器全部保持只读。

选择该路线是因为性能、动画流畅度、低资源占用和桌面集成是当前最高优先级。

## UI 方向

宠物应支持移动放置，也可以选择常驻顶部。

第一版紧凑 UI 只展示少量高信号指标：

- 今日 token 数。
- 今日请求数。
- 本周 token 数。
- 成功率或平均延迟。

动画状态可以反映：

- 空闲。
- 请求活跃。
- token 增长较快。
- 失败率升高。
- 来源读取问题或统计冲突。

## 暂缓想法

- 完整详细观测 dashboard。
- 本地 Web 面板。
- 更广泛的第三方工具自动发现。
- 预算控制或治理行为。
- 源数据修复或修改；除非后续被明确重新定义，否则仍然不在范围内。
