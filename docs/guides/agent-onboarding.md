# Agent 接入指南

本指南面向后续进入 TallyClaw 仓库的 AI Agent 或开发者。

## 先读这里

1. 阅读 `README.md`。
2. 阅读 `docs/architecture/technical-plan.md`。
3. 查看 `CHANGELOG.md`。
4. 编辑前先执行 `git status --short --branch`。

## 工作规则

- 与项目所有者协作时，默认使用中文。
- 项目文档默认使用中文维护；必要的技术名词、路径、模块名、字段名和代码标识可以保留英文。
- 回答优先采用“先结论、后证据路径、再给具体下一步”的顺序。
- 严格维护对 cockpit tools、`local-ai-gateway` 和未来源系统的只读边界。
- 绝不写入源工具或网关的数据文件。
- 除非得到明确授权且确有必要，否则绝不刷新、轮换、修复或查看凭据内容。
- 保持根目录整洁，不为每次讨论、阶段性实现或临时设计资源新增记录文档。
- 功能迭代、问题修复、重要结构变更或安全边界变化，只需简要更新按日期降序维护的 `CHANGELOG.md`。
- 只有长期有效的架构信息才更新 `docs/architecture/` 或 `docs/decisions/`。

## Agent 工作流

进入项目时：

1. 先继承 README、技术方案和 CHANGELOG，不直接实施。
2. 检查 git 状态，识别未提交或非自己产生的改动。
3. 确认最近进展、下一步、风险和安全边界。
4. 如涉及外部数据源，先确认只读读取方式，再做任何实现。

退出项目前：

1. 说明完成了什么、没有完成什么。
2. 必要时更新 `CHANGELOG.md`。
3. 给出后续可执行任务，避免只留下笼统建议。

## 文档习惯

修改项目时，按以下轻量规则维护文档：

- 用户可见变更、功能迭代、问题修复、重要结构变更或安全边界变化：更新 `CHANGELOG.md`，并将最新日期放在最上方。
- 长期有效的架构或能力边界变化：更新 `docs/architecture/`。
- 有明确取舍且后续需要理解原因的长期决策：更新 `docs/decisions/`。
- 不再新增 `docs/records/`、迭代日志、讨论归档或运维状态文档。

## 安全提醒

TallyClaw 只做观测，不治理、不修复、不修改上游系统。
