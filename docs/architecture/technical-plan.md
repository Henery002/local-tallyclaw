# TallyClaw 技术方案

日期：2026-05-08  
状态：草案基线

## 摘要

TallyClaw 是一个本地 macOS 桌面宠物，用于观测本机既有工具和网关产生的 API token 使用情况。它必须对上游系统保持只读，使用项目自有账本持久化 token 统计，并通过紧凑、动态的桌面 UI 提供日常观测入口。

推荐架构优先考虑性能、动画流畅度、低资源占用和安全边界：

- 使用 SwiftUI 加 AppKit 窗口控制实现原生 macOS 应用。
- 使用 SpriteKit 或轻量帧动画实现像素宠物动效。
- 为 cockpit tools、`local-ai-gateway` 和后续可选工具提供只读数据源适配器。
- 使用独立 SQLite 账本进行长期本地持久化。
- 采用保守去重策略，保留来源证据和统计置信度。

## 强制安全边界

TallyClaw 绝不能写入、刷新、修复、迁移、清理或以任何方式修改上游工具或网关数据。

这包括：

- 不写入 cockpit tools 的数据库、日志、配置文件、账号文件或 token 文件。
- 不刷新 cockpit tools 中 Codex 账号的 `refresh_token` 或相关凭据材料。
- 不写入 `local-ai-gateway` 的数据库、日志、配置或运行态文件。
- 不删除、压缩、迁移或清理源数据文件。
- 即使源记录看起来不一致，也不自动修复源记录。

TallyClaw 只能读取已批准的本地源文件或本地只读 API，然后写入自己的项目专属存储。

## 产品目标

- 用低打扰、轻量的桌面宠物展示本机 API token 活动。
- 即使上游源文件后续被删除，也保留已经观测到的长期总量。
- 支持日、周、月和永久总量统计。
- 让 token 统计可追溯，而不是假装所有来源口径天然一致。
- 避免生成大量文件，保持仓库目录和运行时目录整洁。

## 第一批支持的数据源

### cockpit tools

优先级：高

TallyClaw 应优先读取 cockpit tools API 服务中已有的统计数据。第一版目标颗粒度参考 cockpit 面板：

- 总请求数。
- 成功和失败请求数。
- 总 token 数。
- 输入和输出 token 数。
- 可用时记录缓存 token 与思考 token。
- 平均延迟。
- 成功率。
- 日、周、月和永久窗口。

适配器必须将 cockpit 数据视为只读数据，不得触发账号刷新流程。

### local-ai-gateway

优先级：高

TallyClaw 应读取既有网关用量记录，并归一化到同一内部账本模型。适配器应尽量保留网关侧稳定来源 ID 和原始用量维度。

### 其他工具

优先级：低

其他本地工具可以在后续通过类似插件的只读适配器接入。除非数据源稳定且读取边界安全，否则不纳入第一版。

## 数据模型方向

使用便于追加写入的本地账本，包含归一化事件和聚合快照。

建议核心字段：

- `source_tool`
- `source_kind`
- `source_event_id`
- `source_path_or_endpoint`
- `observed_at`
- `occurred_at`
- `provider`
- `channel`
- `model`
- `request_status`
- `request_count`
- `input_tokens`
- `output_tokens`
- `cache_tokens`
- `thinking_tokens`
- `total_tokens`
- `latency_ms`
- `fingerprint`
- `confidence`
- `dedupe_status`

置信度示例：

- `exact`：用量来自源系统 usage 字段或可信源聚合。
- `estimated`：用量来自推断或估算。
- `partial`：仅有部分维度可用。
- `conflict`：多个来源记录互相冲突。

## 持久化策略

使用一个项目自有 SQLite 数据库保存长期本地状态。运行期间可接受 WAL/SHM 这类临时 sidecar 文件，但系统应定期 checkpoint，避免产生大量数据库分片。

持久化内容应区分：

- 需要追溯时保留的归一化原始观测。
- 日、周、月和永久窗口的已确认聚合计数。
- 数据源同步游标和导入 checkpoint。
- 去重 fingerprint 与冲突标记。

永久总量不能因为上游源文件被删除或轮转而减少。

## 去重策略

采用保守去重。

优先身份顺序：

1. 稳定的上游 request ID 或 event ID。
2. 来源特定事件游标加时间戳。
3. 由来源、模型、token 值、延迟、状态和时间窗口推导出的 fingerprint。

当一条记录可能重复但无法证明重复时，应将其保留为重复候选，而不是静默丢弃。聚合结果应区分高置信总量，以及包含估算或冲突记录的总量。

## 运行时架构

推荐模块：

- `app`：应用生命周期、权限、启动行为和设置外壳。
- `core`：共享领域类型和时间窗口逻辑。
- `data-sources`：只读适配器与来源发现。
- `ledger`：SQLite 持久化、聚合、去重和导入 checkpoint。
- `ui`：宠物窗口、动画状态、紧凑统计界面和交互。

数据流：

```text
只读数据源适配器
  -> 归一化观测记录
  -> 去重与置信度分类
  -> TallyClaw SQLite 账本
  -> 聚合快照
  -> 宠物状态模型
  -> 浮动桌面 UI
```

## 桌面宠物 UI

第一版 UI 应保持克制和紧凑。

窗口行为：

- 透明、无边框浮动窗口。
- 可在桌面任意拖动。
- 可选常驻顶部模式。
- 默认尺寸低打扰。

宠物状态：

- 空闲：轻微呼吸和眨眼。
- 请求活跃：眼睛或身体短暂脉冲。
- token 增长较快：动画更忙碌，并出现短暂活动标记。
- 失败率升高：警觉表情或小型告警标记。
- 来源冲突或读取问题：克制的状态提示。

紧凑统计：

- 今日 token 数。
- 今日请求数。
- 本周 token 数。
- 成功率或平均延迟。
- 可选展示当前最高来源或模型。

详细 dashboard 可推迟到后续原生面板或本地 Web 面板。

## 资源预算

第一版实现目标：

- 空闲 CPU 占用低。
- 内存占用小。
- 使用增量读取，避免高频全量扫描。
- SQLite 批量写入。
- 默认不存储 prompt 或 response 内容。
- 最小化运行时生成文件。

## 能力边界

第一版可以合理支持：

- 只读观测已批准来源。
- 持久化永久计数。
- 日、周、月和永久用量窗口。
- 基础来源、模型、渠道拆分。
- 紧凑的动态宠物反馈。
- 可追溯来源证据。

第一版不应承诺：

- 覆盖所有本机 API 客户端。
- 跨 provider 的完美 token 账务统计。
- 自动修复源数据。
- 迁移上游数据库。
- 处理或刷新凭据。

## 待确认问题

- cockpit tools 数据文件或 API 的精确位置与安全读取方式。
- `local-ai-gateway` 统计 schema 与稳定标识。
- 第一版 UI 使用 SpriteKit 还是更简单的帧动画。
- 聚合计数持久化后，归一化原始观测的保留策略。
- 后续详细观测面板应采用原生面板还是本地 Web。
